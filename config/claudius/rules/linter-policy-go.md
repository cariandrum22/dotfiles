## Linter and Formatter Configuration Policy - Go

### Required Tools

1. **gofmt/goimports** (REQUIRED - Formatter)
   - `gofmt`: Built-in formatter
   - `goimports`: gofmt + import organization (preferred)
   - Run: `goimports -w .`
   - Install: `go install golang.org/x/tools/cmd/goimports@latest`

2. **golangci-lint** (REQUIRED - Meta Linter)
   - Runs 100+ linters in parallel
   - Run: `golangci-lint run`
   - Install: `go install github.com/golangci/golangci-lint/cmd/golangci-lint@latest`

3. **staticcheck** (INCLUDED in golangci-lint)
   - Advanced static analysis
   - Already included in golangci-lint

4. **gopls** (REQUIRED - Language Server)
   - Official Go language server
   - Install: `go install golang.org/x/tools/gopls@latest`

### Configuration

Create `.golangci.yml`:
```yaml
run:
  timeout: 5m
  tests: true
  skip-dirs:
    - vendor
    - testdata

linters:
  enable-all: true
  disable:
    - exhaustruct    # Too strict for most projects
    - depguard       # Unnecessary dependency restrictions
    - wsl            # Whitespace linter too pedantic
    - nlreturn       # Too opinionated about newlines
    - gofumpt        # More strict than gofmt
    - gomnd          # Magic numbers can be acceptable
    - godox          # TODO/FIXME comments are useful

linters-settings:
  gocyclo:
    min-complexity: 15
  
  dupl:
    threshold: 100
  
  goconst:
    min-len: 3
    min-occurrences: 3
  
  misspell:
    locale: US
  
  lll:
    line-length: 120
  
  goimports:
    local-prefixes: github.com/yourorg/yourproject
  
  gocritic:
    enabled-tags:
      - diagnostic
      - experimental
      - opinionated
      - performance
      - style
  
  govet:
    check-shadowing: true
    enable-all: true

issues:
  exclude-rules:
    - path: _test\.go
      linters:
        - dupl
        - gosec
        - goconst

output:
  format: colored-line-number
  print-issued-lines: true
  print-linter-name: true
```

### Makefile Integration

Create `Makefile`:
```makefile
.PHONY: fmt lint test check

fmt:
	goimports -w .
	go mod tidy

lint:
	golangci-lint run

test:
	go test -race -coverprofile=coverage.out ./...
	go tool cover -html=coverage.out -o coverage.html

check: fmt lint test

# Quick check before commit
pre-commit: fmt
	golangci-lint run --fix
```

  run: go test -race -coverprofile=coverage.out ./...

### Programming Paradigm Note

Go has **limited functional support** by design. The language favors **clear, procedural code**:

1. **Why Go Prefers Procedural Style**:
   - Designed for simplicity and readability
   - Explicit error handling over monadic patterns
   - No generics until Go 1.18 (still limited)
   - Goroutines and channels for concurrency over functional abstractions
   - "A little copying is better than a little dependency"

2. **Idiomatic Go Patterns**:
   ```go
   // Good: Explicit error handling
   result, err := doSomething()
   if err != nil {
       return nil, fmt.Errorf("failed to do something: %w", err)
   }
   
   // Good: Simple for loops
   for i := 0; i < len(items); i++ {
       process(items[i])
   }
   
   // Good: Early returns
   if condition {
       return earlyResult
   }
   // Continue with main logic
   ```

3. **Limited Functional Features**:
   ```go
   // Acceptable: Simple anonymous functions
   sort.Slice(people, func(i, j int) bool {
       return people[i].Age < people[j].Age
   })
   
   // Acceptable: Method chaining with interfaces
   result := strings.NewReader(input).
       Read(buffer)
   
   // Avoid: Complex functional abstractions
   // Go lacks map/filter/reduce - use loops instead
   ```

4. **When to Use Functional Patterns in Go**:
   - Simple callbacks and handlers
   - Sorting with custom comparators
   - HTTP middleware chains
   - Options pattern for configuration

5. **Prefer Go's Idiomatic Patterns**:
   - Named return values for clarity
   - Table-driven tests
   - Explicit over implicit
   - Flat over nested
   - Composition over inheritance
   - Errors are values

**Recommendation**: Write Go code that looks like Go, not like Haskell or Scala. The language's simplicity is a feature, not a limitation.
